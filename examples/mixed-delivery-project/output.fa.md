# مثال: پروژه مختلط — خروجی (فارسی)

این یک مثال ساختگی از مدیریت دلیوری دوزبانه است — زمینه کسب‌وکار فارسی، شناسه‌های فنی انگلیسی.

---

## تشخیص پروژه: بازطراحی فرآیند onboarding مشتری

**وضعیت:** 🟡 زرد

مهندسی آماده شروع است اما منتظر تأیید دیزاین است. قرارداد API بین frontend و backend نهایی نشده که ریسک واگرایی در توسعه موازی ایجاد می‌کند. backlog وجود دارد اما اولویت‌بندی نشده که مانع تعیین timeline واقعی می‌شود.

---

## مسیر بحرانی (Critical path)

۱. دریافت تأیید دیزاین ← شروع مهندسی — قبل از جلسه ذینفعان (این هفته)
۲. نهایی شدن API contract (frontend ↔ backend) — روز اول مهندسی
۳. اولویت‌بندی backlog (آیتم‌های Must have تأیید شوند) — قبل از sprint ۱
۴. تکمیل sprint ۱ — هفته ۲
۵. تعریف معیار و برنامه rollback — هفته ۲
۶. بررسی پیشرفت توسط ذینفعان — هفته ۳
۷. ارزیابی آمادگی لانچ — بر اساس سرعت sprint مشخص می‌شود

---

## ریسک‌های اصلی

| ریسک | احتمال | تأثیر | اقدام کاهشی | مالک |
| --- | --- | --- | --- | --- |
| نهایی نشدن API contract — واگرایی frontend/backend | بالا | بالا | API contract را به عنوان خروجی روز اول تعریف کن. شروع مهندسی را تا تأیید بلاک کن. | لید فنی |
| تأخیر در تأیید دیزاین | متوسط | بالا | timeline بررسی دیزاین را امروز تأیید کن. اگر تا پایان هفته کامل نشد اسکالیشن کن. | PM |
| بدون rollback plan — اولین ریلیز به production محافظت نشده است | بالا | متوسط | معیار rollback را قبل از هر ریلیز به production تعریف کن. | لید مهندسی |
| backlog اولویت‌بندی نشده — ریسک ساختن چیز اشتباه اول | بالا | متوسط | قبل از sprint ۱ با تیم اولویت‌بندی کن. | PM |

## اقدامات فوری

| اقدام | مالک | تا |
| --- | --- | --- |
| تأیید timeline بررسی دیزاین و تاریخ sign-off | PM | امروز |
| نهایی کردن API contract (frontend ↔ backend) | لید فنی | روز اول kickoff مهندسی |
| اولویت‌بندی backlog — آیتم‌های Must have تأیید شوند | PM | قبل از جلسه ذینفعان |

| هدف | وضعیت فعلی | هدف | baseline تأیید شده؟ |
| --- | --- | --- | --- |
