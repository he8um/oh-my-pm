# Examples

This directory contains small, deterministic examples for using OH MY PM packages together.

The examples are private workspace examples. They do not publish a package and they do not create a binary.

Current examples:

- status through the CLI and injected Runtime
- doctor through the CLI and injected Runtime
- plan through the CLI, Runtime, Planner, local Provider, and deterministic Skills
- installer dry-run planning through in-memory adapters
- controlled installer execution through explicit write adapters
- rollback capture examples
- installer package assembly dry-run through in-memory adapters
- installer archive planning from assembly dry-run results
- installer signed release metadata from archive plans
- installer release integrity verification of metadata against archive plans
- installer release channel metadata grouping verified releases locally
- installer local update policy evaluation against an installed manifest
- installer update impact preview comparing current files with candidate entries
- installer rollback impact preview comparing current files with backup entries
- installer decision report aggregating local preview layers into one verdict
- installer audit event model describing the local preview pipeline in memory
- installer audit trail export plan rendering audit events as an in-memory payload
- installer guarded write capability evaluating whether writes would be allowed under an explicit policy
- installer explicit write approval token binding an intent, root, and decision value
- installer explicit write execution plan listing local step kinds for install, update, or rollback
- installer write execution confirmation checklist reporting pre-write readiness
- installer controlled write adapter contract validating declared adapter capabilities
- installer controlled write execution dry-run envelope aggregating every write readiness layer
- installer release readiness summary aggregating local preview readiness into one report
- installer v0 release candidate checklist evaluating whether the repository is a v0 candidate
- installer public v0 release notes draft rendering public-safe notes from local readiness reports
- installer guarded release artifact plan reporting which release outputs would be planned (creation disabled)
- installer guarded local artifact assembly dry-run envelope aggregating release readiness layers (creation disabled)
- installer guarded artifact creation permission evaluation over the assembly envelope (creation disabled)
- installer local artifact creation execution plan sequencing future creation steps (creation disabled)

Installer examples do not add a CLI install command and do not package or download release artifacts. The assembly example builds a manifest only, and the archive plan example plans a name, checksum, and entries only; no archive file is created and nothing is published. The signed metadata example uses a deterministic placeholder signature only: no signing keys exist and nothing is signed for real. The integrity example checks consistency between metadata, archive plan, and the placeholder signature shape only; it does not verify real cryptographic signatures. The channel example builds local metadata only: there are no remote locations and nothing leaves the machine. The update policy example returns a local eligibility decision only; it retrieves nothing and executes no installation. The update impact example compares provided current files with candidate entries and reports create/replace/remove/unchanged counts only; it touches no files. The rollback impact example compares current files with backup entries and reports restore/remove/missing/unchanged counts only; it executes no rollback. The decision report example aggregates every local preview layer into one ready/blocked/review-required verdict and renders it as markdown; it is report-only and executes no install or rollback. The audit event example turns the decision report into a deterministic in-memory event sequence (preview start, section evaluation, decision, completion) and renders it as markdown; it is event-model only, with no persistence and no telemetry — nothing is written, logged, or sent. The audit trail export example renders that event sequence into an in-memory export payload (JSONL) and reports event count, byte size, and a deterministic fingerprint only; the export payload is in-memory only and is not persisted or sent, and the raw content never leaves the example. The guarded write capability example evaluates whether a write-capable install would be allowed under the default preview-only policy; the result is blocked, and it is capability-model only — nothing is written, executed, retrieved, logged, or sent, and there is no write adapter, command, or remote field. The explicit write approval token example builds a deterministic, non-secret token that binds an intent, root, and decision value and validates it; the token is descriptive text only — it carries no secret, key, signature, or timestamp, and nothing is written, executed, retrieved, logged, or sent. The explicit write execution plan example turns an allowed capability decision plus the existing local preview data into a deterministic list of planned write steps (step kind and path) for the install intent; it is planning only — there is no content, write adapter, command, destination, or execution-result field, and nothing is written or executed. The write execution confirmation checklist example builds a deterministic checklist over the capability and execution plan (intent consistency, decision readiness, capability, plan readiness, step presence); it is confirmation-only — it reports pre-write readiness and executes nothing. The controlled write adapter contract example validates a declared adapter metadata contract (name, capabilities, approval requirement, rollback-capture support) against the confirmation checklist and execution plan; it is metadata-only — no adapter is constructed or called, and nothing is written or executed. The controlled write execution dry-run envelope example aggregates every write readiness layer (capability, approval token, execution plan, confirmation checklist, adapter contract) into one non-mutating inspection object; it reports readiness only — no adapter is called and nothing is written or executed. The installer release readiness summary example aggregates the decision report, audit trail export dry-run, and controlled write dry-run envelope into one release-readiness report and renders it as markdown; it is summary-only — no release artifact is created and nothing is written or executed. The v0 release candidate checklist example builds a deterministic 14-item checklist from aggregated local readiness and public-hygiene signals and renders it as markdown; it is checklist-only — it decides candidacy without creating any release, and nothing is written or executed. The public v0 release notes draft example renders public-safe, summary-only release notes (status, included/safety/not-included/validation/next sections) from the checklist and release-readiness reports and renders them as markdown; it is documentation-only — no GitHub release, tag, or artifact is created, and nothing is written or executed. The guarded release artifact plan example reports which release outputs (release notes, package manifest, archive plan, release metadata, integrity metadata, channel metadata) would be planned from the local dry-runs and keeps `creationAllowed` false; it is planning-only — nothing is created, written, or executed. The guarded local artifact assembly dry-run envelope example aggregates the guarded release artifact plan and the package assembly, archive, metadata, integrity, and channel dry-runs into one readiness envelope with per-layer readiness flags and keeps `creationAllowed` false; it is readiness-only — nothing is created, written, or executed. The guarded artifact creation permission example evaluates whether a future explicitly-enabled local artifact creation phase would be permitted (explicit mode, ready assembly envelope, explicit approval) and keeps `creationAllowed` false even when permission is allowed; it is evaluation-only — creation stays disabled and nothing is created, written, or executed. The local artifact creation execution plan example sequences the ordered creation steps a future explicitly-enabled phase would take (one prepare step per planned release output) from the permission, artifact plan, and assembly dry-runs and keeps `creationAllowed` false; it is planning-only — no step is executed and nothing is created, written, or executed.

The example Kernel is an injected deterministic boundary used only for examples, demonstrating dependency injection. The private CLI wrapper now uses the real WASM Kernel binding instead.
