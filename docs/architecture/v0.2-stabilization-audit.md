## Executive Summary

v0.2 is architecturally sound. The full baseline is green (contracts, structure,
boundaries, public surface, version consistency, all TS + Rust builds, `pnpm test`,
`pnpm validate`, `pnpm mcp:smoke`, `cargo test`, `cargo clippy -D warnings`, and the
`wasm32-unknown-unknown` kernel build all pass on a clean worktree). The GitHub
provider boundary is tightly enforced (GET-only, single fixed host, single page, no
writes/GraphQL/cache/retry/sync), the runtime provenance allow-list discards raw
provider data, CLI and MCP converge on shared resolvers, and the release/installer
transaction is layered and network-free.

No `critical` or `high` **behavioral** defect was found. The only confirmed defects
are **documentation drift**: three stale "eight-tool" code comments and five stale
doc statements (two tool counts, three configurable-field lists, one stale
"No comments" limitation). All were corrected in this commit — comment/doc-only, no
executable behavior changed, so no regression test is required (the corrected
comments annotate ten-entry arrays that are already asserted by existing tests).

The remaining findings are maintenance debt (duplicated numeric domain constants
that hold identical values today) and asymmetric cross-platform CI coverage
(source-removal and prefix-relocation exercised on POSIX only). These are `medium`
/`low`, not release blockers, and are recorded as deferred with recommendations.

**Readiness:** v0.2 is ready for another **bounded** feature. Selected next action: **A**
(proceed to bounded PR reviews / review comments) — see §15.

> **Post-audit update.** The two follow-up items this summary flagged are now closed:
> asymmetric cross-platform CI coverage (F-TEST-1) is resolved by the
> `windows-release-install` source-removal/prefix-relocation steps, and the
> duplicated-constant debt (F-DUP-1) is resolved by centralizing the canonical list
> limit constants. RC readiness remains a deliberate decision, not automatic.

## Module Boundary Findings

Dependency direction in `providers/src/github/` is acyclic and one-way:
`provider.ts` → {`query.ts`, `normalize.ts`, `transport.ts`, `constants.ts`,
`types.ts`}; `selection.ts` → `query.ts`; `query.ts` → `types.ts`; `normalize.ts` →
`constants.ts` + contracts; `transport.ts` → `constants.ts` + `types.ts`
(`provider.ts:26-52`, `selection.ts:11-18`, `query.ts:6-15`, `transport.ts:9-14`).
The source/state/kind unions are duplicated in `types.ts:37-41` **deliberately** to
keep `query.ts` free of a dependency on `selection.ts` (documented at
`types.ts:37-41`).

Concern separation holds and is documented per-module: query = pure parse/build
(`query.ts:1-4`), selection = pure override/default resolution (`selection.ts:1-8`),
normalize = pure validation/projection (`normalize.ts:1-5`), transport = sole network
module (`transport.ts:1-7`), provider = orchestration only (`provider.ts:1-4`).

### F-BND-1 — provider.ts peeks at the normalizer's `data` shape
- **severity:** informational · **status:** intentional
- **affected files:** `providers/src/github/provider.ts:346-354`, `:395-399`

- **recommended action:** none required. Optionally consolidate PR-detection into one
  exported predicate if it drifts. **code change required now: no**

### F-DUP-1 — limit `1..100` and default `50` declared independently in 5+/3 sites
- **severity:** low · **status:** resolved
- **affected files (before):** `parser.ts` (list/comment/review min/max literals),
  `selection.ts` (inline `1..100` literal), `settings.ts`, `config.ts`,
  `constants.ts` (max/default 50)
- **evidence (before):** `100` was declared as an independent constant in five layers
  and inlined in `selection.ts`; the default `50` in three. Values were **identical
  everywhere**, so there was no correctness defect — only a single-source-of-truth gap.
- **resolution:** `constants.ts` now declares the canonical list bounds
  `GITHUB_MIN_LIMIT` (1), `GITHUB_DEFAULT_LIMIT` (50), `GITHUB_MAX_LIMIT` (100), exported
  through the provider barrels. `config.ts`, `settings.ts`, `selection.ts`, and the CLI
  `parser.ts` import and use them (the CLI parser also imports the already-canonical
  `MIN/MAX_GITHUB_COMMENT_LIMIT`, `MIN/MAX_GITHUB_REVIEW_LIMIT`, and
  `MIN/MAX_GITHUB_REVIEW_COMMENT_LIMIT` instead of re-declaring them). The public
  compatibility aliases `DEFAULT_GITHUB_PROVIDER_LIMIT` and `GITHUB_CLI_DEFAULT_LIMIT`
  are retained and resolve to `GITHUB_DEFAULT_LIMIT`. A `validate-boundaries.mjs`
  invariant now forbids the executable config/settings/selection/CLI layers from
  re-declaring these numeric bounds. The MCP JSON schemas and release/install verifiers
  intentionally keep their independent literals as repository-independent boundary-defense
  assertions and were **not** changed. No numeric value or behavior changed.
  **code change required now: done (behavior-preserving refactor)**

### F-DUP-2 — forward-looking `PROVIDER_ID_VALUES` wider than enforced set
- **severity:** informational · **status:** intentional
- **affected files:** `contracts/generated/ts/providers.ts:7`, `providers/src/config.ts`
  allow-list, `runtime/src/plan-utils.ts:121`
- **evidence:** the contract enum lists seven providers
  (`local, github, clickup, airtable, linear, jira, notion`); only `local` + `github`
  are registered/allowed. Superset, not a conflict; the runtime allow-list gates
  against the contract enum.
- **impact:** the contract enum is not a reliable source of the *currently supported*
  provider set — a reader must consult the config allow-list.
- **recommended action:** none required; note in docs if it causes confusion.
  **code change required now: no**

## Runtime and Provider Flow

**Provenance (verified):** raw provider data and unknown nested fields are discarded —
each field is individually extracted with type guards and `source`/`type` are
enum-validated (`plan-utils.ts:157-221`). The MCP source projection never exposes
comment bodies: `McpGitHubComment` (`github-tool-runner.ts:137-143`) has no `body`
field and the projection never reads `data["body"]` for a comment
(`github-tool-runner.ts:195-204`; comments capped at 50, `:232-237`).

**Provider boundary (verified — all negatives confirmed by grep + code):**
- GET-only, enforced twice: provider always sends `method:"GET"` (`provider.ts:153`);
  transport rejects non-GET (`transport.ts:132-134`); type pins the literal
  (`types.ts:6`).
- Single host: `GITHUB_API_ORIGIN`/`GITHUB_API_HOSTNAME` (`constants.ts:6,26`);
  same-origin enforced on the initial URL and every redirect (`transport.ts:43-45`,
  `:142`, `:182`).
- Fixed API version `2026-03-10` (`constants.ts:7`; sent `provider.ts:71`,
  `transport.ts:121`).
- One page only (`page=1` at `provider.ts:217,285,365`); no pagination loop; the only
  `for(;;)` is redirect handling (`transport.ts:150-193`).
- No retries (single `fetchImpl`, `transport.ts:155`; throws on catch `:161-162`);
  no cache; no timers except the abort timeout (`transport.ts:152`, cleared `:164`);
  no writes (0 POST/PUT/PATCH/DELETE); no GraphQL; no origin override
  (module consts, no setter/env); no token persistence (closure var only,
  `transport.ts:115`; never written to fs).

**Maximum HTTP request counts per operation (verified from `provider.ts` code paths):**

| Operation | Max requests | Path |

| overview | 2 | repo GET + issues list GET (only when `limit>1`) — `provider.ts:254,278,286` |
| repository | 1 | `fetchRepositoryRecord` — `:312-316` (selection forces `limit:1`, `selection.ts:330-331`) |
| issues (list) | 1 | `runIssueSearch` → `/search/issues` — `:319,218` |

| search | 1 | `runIssueSearch` → `/search/issues` — `:333,218` |
| item (issue), no comments | 1 | GET issue — `:390` |

## CLI and MCP Parity

Both callers converge on the same shared primitives —
`resolveGitHubProviderSettings`, `resolveGitHubSourceSelection` (`selection.ts:147`),
`createGitHubRuntimeRequest` (`request.ts:25`) — so repository resolution,
source-selection defaults, validation ranges, and applicability rules are identical
by construction; only the request `caller` tag differs (`request.ts:48,53`). All
workflows and parameters (overview/repository/issues/pull-requests/item/search;
state/kind/limit/number/query/includeComments/commentLimit) share the same ranges:
limit 1..100 default 50, comment limit 1..50 default 20, number ≥1, query trimmed
≤256 no control chars.

MCP cannot supply a config path or token: `githubInputShape` (`server.ts:444-496`)
has only workflow fields; MCP resolves config from env/OS-standard location with no
`explicitPath` (`github-tool-runner.ts:119-134`, `:116-118`); token is read only from
env at the call boundary (`:318-319`). CLI intentionally supports `--provider-config`
(`parser.ts:112-118`); it never accepts a token option (`parser.ts:90-91`). MCP
startup and `tools/list` are network-free (`server.ts:800-801,864`,
`startOhMyPmMcpStdioServer` `:918-922`). MCP errors are sanitized
(`server.ts:698-727,879-907`); structured outputs are strict-projected.

### F-PAR-2 — CLI exit codes not centralized
- **severity:** low · **status:** deferred

  `cli/src/local-process.ts` (multiple), `distribution/bin/oh-my-pm-install.mjs`

## Skill-Layer Consistency

`project-signals.ts` is cohesive: a single responsibility (pure, deterministic
extraction of risk and next-task candidates from explicit/Markdown/GitHub inputs), no
Node/fs/network/clock access, injected `now` (`parseComparableInstant` is clock-free).
Classification is mutually exclusive by early-`continue` dispatch
(`extractRiskCandidates:1115-1128`, `extractNextTaskCandidates:1190-1198`): comment
(`isGitHubComment:339-345`) → repository (`isRepositoryRecord:291-297`) → issue/PR.
Repositories never become tasks (`classifyGitHubNextTask` returns null for a record,
`:639`); comments never become top-level issues (routed to comment-only classifiers,
`:399-431,469-502`); arbitrary prose is not classified (only recognized headings /
markers count, `:394-398,463-468`).

Checked tasks are excluded (`:989,1055,423,493`); terminal/parent statuses respected
(`:565,645-647`; comment tasks require open/draft parent, `:348-351,471`). Limits:
`MAX_RISKS = 20` (`:51`, applied `:1154`), `MAX_NEXT_TASKS = 10` (`:52`, applied
`:1220`), handoff `MAX_SECTION_ITEMS = 5` (`create-handoff.ts:19`, enforced
`:69,77,214`). Comments do not affect status counts/highlights
(`summarize-status.ts:51` filters `!isGitHubCommentItem`); author never replaces owner
(distinct fields; `commentMetadata` sets author, omits owner, `:353-360`). Ordering
and dedupe are deterministic (insertion order + explicit high/medium/low bucketing
`:1199-1203`; dedupe by id/sourceId/markdown-key preserving first occurrence,
`OrderedRisks`/`OrderedTasks` `:905-947`; no `Date.now`/random/locale sort). Persian
and English aliases are bounded fixed `const` sets with deterministic normalization
(`:688-728,766-797,752-764,812-850,150-180,71-91`).

## Release and Installer Stability

The install core (`release-install-core.mjs`) is the single mutation authority
(`:900`), depends on nothing in the repo (`:1-9`), and is network-free (RELEASE.json
`installer.network:false`, enforced `check-release-bundle.mjs:242`,
`release-install-core.mjs:385`; source scan bans network markers,
`release-bundle-utils.mjs:677-709`).

- **Flat/hoisted, symlink-free bundle:** forced via `--config.node-linker=hoisted`
  (`release-bundle-utils.mjs:158-171`); symlink-escape/dangle scans
  (`:572-588,601-602`); install-time re-checks (`release-install-core.mjs:406-425`,
  `:868-893`).
- **Complete kernel WASM assets:** three assets defined once (`release-bundle-utils.mjs:44-48`),
  staged deterministically (`:189-340`), completeness-gated (`:818-824`), re-verified in
  both verifiers (`check-release-bundle.mjs:360-401`, `check-release-install.mjs:238-246`).
- **Platform-aware POSIX shim policy:** `requiresPosixShimExecutableMode(platform)`
  (`release-install-core.mjs:249-251`); verifiers skip the exec-bit on Windows
  (`check-release-bundle.mjs:350-354`, `check-release-install.mjs:263-267`). See
  memory note *release-installer-platform-policy*.
- **No-shell Windows runtime invocation:** installed command launches
  `node <installed .mjs>` directly on Windows, shim on POSIX
  (`check-release-install.mjs:131-136`), via `execFileSync`/`StdioClientTransport` (no
  shell). Rationale cites CVE-2024-27980. See memory note *windows-cmd-shim-spawn*.
  (The build-time `pnpm deploy` deliberately uses `shell:true` on Windows because the
  pnpm launcher itself is a `.cmd` — a separate concern.)
- **Exactly four shims validated:** `oh-my-pm`, `oh-my-pm.cmd`, `oh-my-pm-mcp`,
  `oh-my-pm-mcp.cmd` in all three layers (`release-install-core.mjs:516-525`,
  `check-release-install.mjs:252-262`, `.github/workflows/ci.yml:218-221`).
- **Transactional apply + rollback:** revalidate → stage into `.install-tx-<pid>` →
  backup → rename → post-install verify → `restoreBackups()` on failure
  (`release-install-core.mjs:900-1126`).
- **Source-removal + prefix-relocation:** verified — POSIX CI only (see F-TEST-1).

## Error Taxonomy

Structured `OMP-<layer>-<n>` codes are namespaced and unique (verified by grep):
K-1xxx (kernel), R-2xxx (runtime), C-3xxx (cli), P-4xxx (provider), S-5xxx (skills),
I-6xxx (installer). Provider codes `OMP-P-4001..4009` each map 1:1 to a stable message
(`providers/src/errors.ts:4-25`) and are all reachable
(`provider.ts:110-130,161,222,…`; 4001 is emitted at the registry layer). Failures are
sanitized by construction: `providerFailure` messages are the stable default or fixed
literals; tokens appear only in the `Authorization` header (`transport.ts:125`), never
in a thrown message; response bodies are never placed in errors (`transport.ts:7`
guarantee). Only user-supplied strings (their own root/repo/provider-id) are echoed, by
design (`local-process.ts:396-398`).

Invalid input fails **before** network/token/clock: the CLI github path resolves
config → settings → selection, each failing at exit 2, and only then reads clock,
reads token, and constructs the transport (`local-process.ts:340-394`, comment
`:341-342,384-385`); the MCP path enforces the same order
(`github-tool-runner.ts:265-310`).
