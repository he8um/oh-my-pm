# OH MY PM v0.2.0

Status: Unreleased

Current development version: `0.2.0-alpha.0`

This document tracks the in-progress `v0.2` development line. `0.2.0-alpha.0`
is not published; there is no `v0.2` tag, GitHub Release, or download. `main`
may contain unreleased `0.2.0-alpha.0` work at any time.

## Planned scope

The following are planned for `v0.2` and are **not** complete:

- provider aliases/profiles only if justified by real usage
- richer read-only GitHub context (comments/timelines) as a separate bounded phase

## Already in the development baseline

- development baseline aligned to `0.2.0-alpha.0`
- self-describing version, bundle, and archive verification (the version is
  derived from `version.json`, `RELEASE.json`, or asset filenames rather than
  hard-coded)
- preview-first self-installation from an extracted verified release bundle:
  every portable bundle ships `bin/oh-my-pm-install.mjs`, which installs a
  versioned, source-independent copy under an explicit `--prefix`. Preview is
  the default, `--apply` is required to write, and `--force` replaces only the
  exact managed targets. Implemented in the current development line; this does
  **not** publish a `v0.2` release, tag, or download.
- a strictly read-only, explicitly opt-in GitHub provider for repository
  metadata, issues, and pull requests, exposed through both the CLI (`github
  brief|risks|next|handoff`) and MCP (`github_project_*`). `GET`-only against a
  fixed `api.github.com` origin, REST API version `2026-03-10`, with an optional
  `OH_MY_PM_GITHUB_TOKEN`. Implemented in the current development line; this does
  **not** publish a `v0.2` release, tag, or download.
- asynchronous provider, Runtime, CLI, and MCP execution boundaries to support
  real read-only network providers, with local workflows unchanged and offline.
- source-aware, line-level deterministic risk and next-task extraction for
  Markdown and GitHub context, with English and Persian headings/markers, exact
  GitHub label and status rules, overdue inference from the injected timestamp,
  one risk/task per GitHub item, 20-risk and 10-task limits, and optional public
  provenance metadata. Rule-based only — no LLM, embedding, or fuzzy scorer. See
  [the deterministic extraction guide](../deterministic-extraction.md).
- strict, read-only provider configuration (`providers.json`) with optional
  GitHub `enabled`/`defaultRepository`/`defaultLimit`/`defaultSource`/`defaultState`
  defaults, plus offline `providers status`/`providers doctor` commands and an
  explicitly confirmed, single-request GitHub access diagnostic. The MCP surface
  gains `provider_status` and `github_provider_diagnostics`, for exactly ten
  tools. No secret is ever stored in configuration; the token stays in
  `OH_MY_PM_GITHUB_TOKEN`. See
  [provider configuration](../providers/configuration.md) and
  [provider diagnostics](../providers/diagnostics.md).
- explicit GitHub source selection across CLI and the four GitHub MCP workflow
  tools: `overview`, `repository`, `issues`, `pull-requests`, `item` (single
  issue/PR with type auto-detection), and repository-scoped `search`, each with
  `open`/`closed`/`all` state selection and search `kind` filtering. The default
  stays `overview` + `open`; user search terms can never override the injected
  repository/state/kind scope. `GET`-only, single page, no comments/timelines/
  diffs, no write-back. See
  [GitHub source selection](../providers/github-source-selection.md).

## Guarantees carried forward from v0.1.0

- no project file writes
- no context upload
- no telemetry
- stdio-only MCP transport
- Node.js 20+ runtime; no npm package
- local Markdown project workflows remain fully offline (no network, no token)

## New in the v0.2 development line

- Network access is off by default and scoped: only the explicit GitHub
  command/tool reaches the network, and only over read-only HTTPS `GET` requests
  to `api.github.com`. `RELEASE.json` declares this conditional, opt-in surface.
- Portable bundles ship a complete, prebuilt Node-loadable Kernel binding on
  Ubuntu, macOS, and Windows. Bundle assembly explicitly stages the generated
  JS glue, WASM binary, and CommonJS manifest and verifies them byte-for-byte;
  installation never rebuilds the Kernel, and the generated binding is never
  committed. This closes a Windows-only packaging gap where `pnpm deploy` omitted
  the generated Node WASM glue file.
