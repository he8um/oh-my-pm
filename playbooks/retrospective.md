# Playbook: Retrospective

## Purpose

Extract actionable lessons from a completed sprint, milestone, or project — and make sure they actually change behavior.

## When to use

- A sprint or milestone is complete
- A project has closed (or been cancelled)
- A significant incident or failure occurred and root cause is needed
- A team is repeating the same mistakes and needs structured reflection

## Inputs needed

```txt
What period or milestone does this retro cover?
Team list (who should participate)
Sprint metrics or delivery outcome (on time? complete?)
Any incidents or notable events in this period
Previous retro action items (for follow-up)
```

## Fast-start questions

1. "What was supposed to happen? What actually happened?"
2. "If we had to do this sprint again, what's the first thing we'd change?"
3. "What went smoothly that we should actively preserve next time?"
4. "What systemic issue kept coming up, even if it resolved each time?"
5. "Which previous retro action items did we actually do?"

## Recommended process

1. **Set context** — What period, what goal, what happened.
2. **Review previous retro actions** — Which were completed? Which were not? Why?
3. **What went well** — What worked? Preserve it explicitly.
4. **What to improve** — What did not work? Be specific about root cause.
5. **Action items** — For each improvement: one concrete action, one owner, one date.
6. **Shoutouts** — Who deserves recognition?
7. **Close with next retro date** — When is the follow-up?

## Root cause depth

When something goes wrong, do not stop at the surface:

- Surface: "QA was late."
- One level deeper: "QA started late because dev delivered late."
- Root: "Dev delivered late because acceptance criteria were unclear at sprint start, causing rework."

The action item should address the root, not the surface.

## Output format

```txt
Period: [Sprint / Milestone / Project name]
Date: [Date]
Facilitator: [Name]

Previous action items:
| Action | Owner | Status |

What went well:
- [Item]
- [Item]

What to improve:
- [Issue] — root cause: [Why it happened]

Action items:
| Action | Owner | By |

Shoutouts:
- [Name]: [What they did]

Next retro: [Date]
```

## Action item quality bar

A good action item is:

- Specific: "Add acceptance criteria review to sprint kickoff checklist" not "write better criteria"
- Owned: one person is responsible
- Time-bound: a date, not "soon"
- Systemic: addresses the root cause, not the symptom

A retro with no action items is a retro that changes nothing.

## Timebox guidance

- 60 minutes maximum for a sprint retro
- 90 minutes for a milestone or project retro
- 30 minutes for a lightweight sync retro

Do not let the retro become a blame session. Systemic issues, not individual mistakes.

## Quality checklist

- [ ] Previous retro action items reviewed first
- [ ] At least one "what went well" item
- [ ] Each "what to improve" item has a root cause, not just a symptom
- [ ] At least one concrete action item with owner and date
- [ ] Next retro date is set

## Common mistakes

- Retros that produce no action items
- Action items with no owner or date
- Repeating the same "what to improve" items from previous retros (signals actions are not happening)
- Skipping the "what went well" section (preserving what works is as important as fixing what doesn't)
- Not following up on previous retro actions
